腾讯mini项目-whosbug

Created2022-09-14|Updated2022-09-19
|Post View:

前言

暑假参加了学校与腾讯合作的mini项目，由于当时看见了 whosbug 项目是官方标明的 P1 难度，就想着难得参加腾讯的项目，而且使用的语言是之前就感兴趣的 Golang ，于是就选择了 whosbug

在项目过程中，也结识了给力的队友，给力的老师，给力的导师，从这表面14天，实则一个月的项目中，也确实接触了很多之前一点都不懂的东西，涉及了很多全新的领域，故写一篇博客来记录下

whosbug 概述

whosbug 分为对应的 CI 插件部分和 web 应用端部分，说白了，就是一个用来让你甩锅甩的更准确，更有理有据的程序

站在用户的角度看 whosbug，在你的项目发生报错的后，把对应的报错信息，如第三方的提交项目，出错的版本，以及发生错误的崩溃堆栈，提交到暴露给用户的 web 应用端，然后用户其实就等着出结果就可以了，在用户的视角里，这个提交完之后，web 应用端就会返回一个字符串，即这次报错的责任人，也就是造成这次错误最主要的人，还会对应返回一个小于等于1的浮点数，表示有多大的几率可以相信这次错误确确实实是这个人造成的，这样一次 whosbug 就完事了，即在用户角度，提交出错信息，返回责任人就ok

站在运行的角度，一份代码，或者说一个项目，要先进 CI 进行分析，如果以前没有分析过，那就拉取最新版本进行 antlr 语法分析，如果以前有分析过，那就看看要分析的这个版本是不是这个项目的最新版，是的话就不做操作，不是就拉取最新版，再进行分析，所有的这些分析结果，会生成对应的信息，经过加密后发送到 web 应用端，CI 的使命就结束了

web 应用端这边，最后根据提交的数据，相应的进行置信度更新，然后把新的置信度等信息存进数据库里，web 应用端这边的任务也完成了，整个 whosbug 的除去查询的流程就是这样

而 web 应用端进行责任人的计算部分原理也不复杂，收到用户发送的出错信息，web 应用端就根据项目名，版本号等信息抓取数据库的对应项目的数据，由数据库里的信息返回出错相关的方法，类等等，最后把抓取出的这些信息交到 keyman 算法进行计算，可以得出一个由出错贡献度降序排列的出错函数集，显然第一个数据就是我们需要返回的责任人

前置学习

这个项目作为一门暑期的课程开设，需要面试，这也让我有点紧张，暑假的课程，在5月中就需要进行前置学习，并在学习期间做好笔记，对自己学习的东西，期间遇到的问题，对应进行的思考进行记录

当时要学习的有很多，我基本很多都是从头开始，比如 git 的各种操作，Golang 的基本语法，甚至还学了一点 gin 框架，当时还走了弯路，以为 antlr 是重点，为了看懂 antlr ，直接去现学了一天的 java，但事实上，最后只用到了 git 和 Golang

当时给的学习资料很多，就是去年的 whosbug 的 Github 仓库，当时也是不是很懂，把代码 download 下来之后乱看一通，觉得自己懂了，其实完全不懂

最后在端午假期的最后一天进行线上的笔试，我感觉我写的很烂，但是可能是我运气好，可能是导师比较喜欢我，最后我还是进了，虽然说有6个组，但是在小两百人里面，选出30人，我自己觉得还是比较幸运的，因为我是0基础，但是我认识的很多人其实都懂相关的东西，所以还是，感谢导师的捞人

项目开始

因为导师有点事情，7月末我们才开了第一次会，进行前置学习的进度对接，那时候好像其他组已经开始了一个周了，属于是起步就给爆了，但是后面大家加班也勤，最后其实留了大概4天的时间来测试，debug，排练等等收尾的东西，其实最后还是遵循了导师说的先紧后松的宗旨

由于第一天我就和导师交流了一下，抓了个 CI 插件在 Github 上的版本的虫，第一次会议导师就让我做会议纪要了，我是这样猜的，我也是第一次做这种多人项目，本来开会就挺紧张的，还要做记录，所以那次听的格外认真

那次会议基本就给项目定分工，定方向，定工期，效率还是很高的，不过在那次会之后，我确实没想到我之后会是整个项目的部署佬，也可以叫运维佬，顺便兼职当下测试佬

whosbug 这个项目不是一个新项目，有三组人已经做过这个项目了，我们的任务是在之前的版本上进行优化，所以这个项目要完成的第一个目标就是 run the project，把老的项目跑起来，然后再把 web 应用端和数据库部署到学校买的腾讯云 TKE 服务器上，这个任务肯定是运维来做，所以一开始 run the project 我就做了10天，当然不全是在项目进行的14天里，遇到的问题也不少，期间也咨询了学长，导师，相应地也学会了很多东西，我学到的最有用的，就是查阅对应的官方文档，这个是从导师身上学的，以前我知道有官方文档的存在，但是遇到问题，总是第一时间，想要去 google，想上 github，想去 sof 找答案，但是如果是比较大的公司的核心产品，官方文档一定是很完备的（不像 coding），对应的问题的解决方法，也会很细化地描述，比网上找到的答案好很多，毕竟可能你和你看的解决问题的那篇博客出的错可能是一样的，但是你们的系统可能有区别，配置有区别，甚至他的操作和你的操作也有区别，可能单纯只是报了同一个错误，不一定是同一个原因引发的，所以在这种时候，查阅官方文档，就很重要

run the project

run the project 最后是要上云的，但是上云之前肯定要本地部署下看看有没有 bug，在这期间，web 和 CI 的几个小 bug 也请导师修复了，本地遇到的只有和数据库连不上，后面发现是端口占用问题，最后也解决了，然后就要上云

不过就单单在一开始 run the project 的上云这一步，就遇到了官方文档无法解决的问题——在腾讯云 TKE 集群上拉取以前老版本的 Dockerfile 打的镜像发生了错误，无法拉取，一开始我以为是 Dockerfile 写错了，研究了差不多三个钟头 Dockerfile，各种语句的书写，各种参数，事实证明当时估计也是熬夜干活脑子干傻了，这个是老项目的 Dockerfile，是成功打过镜像的，肯定不会出错，那么就得换一种思路了，腾讯 TKE 集群是我没法控制的，一般最后才考虑是它出错，所以只能是最原始的拉取错误，即纯拉不到镜像，一开始用的是阿里云的镜像仓库，我用私仓登录来拉取不行，试了下公开仓库，也是拉取不到，又在本地试了下，是可以的，又用 DockerHub 试了下，也是一样的情况，那估计就是 腾讯 TKE 集群的问题，具体问题给他密封了，但是还有一个腾讯云的仓库没试，当时就在想会不会是腾讯云的这个 TKE 集群的某个权限没开全，只能拉腾讯云镜像仓库的镜像，结果一换腾讯云，问题就解决了。这种就是官方文档解答不了的问题，需要自己进行对应思考和尝试

拉取镜像失败这个大卡点过去后，后面的部署也变得顺利了不少，一路照着自己的感觉，查阅的对应文档，没有出现特别大的岔子，web 应用端对应的 service，pg 数据库，pg 的 service 都顺利部署，也学会了用 kubectl，用 kubectl 登录到了云端的 pg 数据库，run the project 就完成了

CI 流水线

要制作，或者说部署 CI 流水线之前，首先得明白，CI 流水线是啥

当你修改了某个仓库的部分代码，假设你修改的是一个用来生成镜像的 Dockerfile，这个 Dockerfile 原来的版本是1.0，随后你 commit 了这次修改，这个版本叫做2.0，那么很显然，所有用了1.0的版本的镜像的容器需要重新生成，部署，当然，你修改了其他的东西，不是 Dockerfile 这种直接生效的东西，也是需要的，因为他是把整个项目打成容器，任何一个小改动，都需要重新打一个镜像

类似的这种由于代码更新，而需要进行一些固定的、重复的工作，就可以交给 CI 流水线来做，可以说，CI 就是一种由代码修改并提交作为触发器的，自动化执行重复的工作的一个流程，有了 CI 流水线，可以很大程度降低开发人员和测试人员的工作量，我需要制作的 CI 流水线有两条，一条是以 CI 代码仓库修改为触发条件的打镜像的流水线，一条是自动进行 whosbug 语法分析，并且进行性能测试的流水线

构建的过程不在这篇文章细说，但是其中遇到的问题很有意思，比如这次项目使用的是腾讯的 Coding 平台，里面有些偶发性的 bug，比如我在构建过程中，要分析的仓库，也是需要拉取的，有些仓库是拉不进去 Coding 的，显示导入失败，有些是导入了，而且数据量也不大，结果却连构建都构建不了，点击构建按钮就直接失败，运行都没运行一下，还有比如说，Coding 每次的构建时长也有限制，还改不了，每次跑 2h 就歇菜了，有一些是可以构建完的，但是因为目标仓库的语法太复杂了，2h 分析不完，每次都构建失败

最有意思的几个错误，最大的卡点还是一开始版本的代码，在语法分析流水线上，一开始是连提供的容器都没进去，结果流水线没识别出来，以为语法分析结束了，然后就结束构建，显示构建成功，结果我往数据库一查，连对应的数据都没有

后来发现是在 Coding 上不能直接设 -t 参数，单独设置了 -t 参数直接就会报错，只能在一个专门给你写参数的文本框写，而且 Dockerfile 里其实有连带着进入容器，但是在 Coding 上给吞了，也很奇怪，所以只能在它提供的地方写 -it ，这样才能进去

后面也有一个一样是因为代码和 Coding 里面的 CI 构建的设置冲突而产生的卡点，是 workspace 的问题，即容器进去后的初始工作目录，Coding 会自己给你规定一个，刚好我们老版本的代码规定的 workdir 也是一样的名字，而且初始里面是有东西的，结果就是 Coding处理重名的 workspace ，居然是直接把自己的 workspace 换名，而且还挂了数据卷，把里面的东西给覆盖成空的，这个问题我一个人研究了小两天，只看出一点眉目，最后还是殊途同归，还是向导师求助了，导师看了下，workdir 一改就可以了

项目测试

CI 能成功运行后，每天就跑跑构建，然后 fix bug，直到导师给了新的 task —— 用 pprof 测试代码的性能，出一些量化指标，或者是做些可视化的内容，比如调用时间图，火焰图，关于 pprof 的内容在我的另一篇博客里，也不多赘述

答辩前收尾

最后大家的测试都差不多，就写个自己版本的 web 应用端的 Dockerfile 和 docker-compose.yaml ，还有数据库相关的 yaml 文件搞一搞，最后把新版的东西都给上云了，最后再用新版本进行一次答辩的排练，就大功告成了

项目答辩

最后几天都是双线程，搞得晕头转向，一边测试项目，fix bug，一边制作答辩用的 ppt ，改了大概3天吧，最后出来的效果我自己还是觉得很满意的，也排练了很多次答辩的场景，当时说的是限时 5 min，第一次展示的时候我们6个人一共用了快10 min，属实是有点夸张了，所以到后面，我们直接细分量化每个人的每个 part 的时间，再反复排练，最后把时间都稳定在了五分钟内，最后正式答辩时我们的用时绝对也是最准的，大概是 4 min 45 s

正式答辩那天，坐车去了松日那边，我们给排在了第一个上，准备了半天，开始之前还搞了个小乌龙，展示用的电脑连着的手机热点断了，好在很快连了腾讯的游客网，一切都有惊无险地进行，展示就好像我们之前排练的那样，没有出岔子，但是最后的提问环节，我这个运维佬没什么好问的，没有回答问题，但是项目里的其他人给问住了，最后还得靠导师替我们回答，给我们解围，再次感谢导师，然后也由问题找到了我们项目组的一些缺点，只能说导师们还是很老辣，看东西一针见血，表达又很清晰，就给人一种很厉害，但是又不是那种世外高人的风格，反而听接地气的，复杂的感觉

我们第一组答辩，自然也是第一个下班，展示完下来后，又看了剩下五个项目组的，本来觉得自己很拉，看完又觉得自己可以了，但是还是心很慌，总觉得是矮子里挑高个，希望导师们不会很不满意吧，后来根据老师说的，好像导师们还是很满意的，那就是大团圆结局了

答辩完六点多，和导师一起吃了顿客家菜，好吃！和导师说拜拜就坐车撤退回学校了

后续

回学校后还有一个剩下的任务，就是把相关的文档完善下，这个项目刚结束两天，我就开始准备做另一个项目了，后面导师在问的时候我才 ddl 极速 writing，写完基本就意味着这个项目暂时告一段落了

感想

通篇都是我的复盘，其实也是我的感想，这是我第一次做实打实的团队协作项目，确实也改变了自己的很多观念，比如，之前我总觉得，一个人只要肯学，肯定是全能的，单人开发，只要是自己会的，效率肯定完爆团队开发，因为团队开发在沟通上有很大的花销，但是经过了这次，我发现，团队协作确确实实可以给你不一样的思路，一颗大脑转再快，也没有火花，一定得多个大脑碰撞才能产生火花，而且你或许可以是 full stack，但是比如说，这个项目需要 6个人，6 个你开发，肯定不如6个擅长各自领域的人开发，你可以很厉害，各种都懂一点，但还是得有个主打方向

又如在上面提过的，查阅官方文档的习惯，我也逐渐在这次项目里养成，还有问问题的习惯，这个也是很重要的，以前总觉得自己厉害，遇到问题，自己查自己解决，一切自助，先不说光靠一个人不可能解决一切问题，就算你解决了，可能花费的时间是别人提点你用的时间的无数倍，你如果是在学习阶段的话，那这种钻研很深，花时间研究是 ok 的，但是在项目开发中，这就是浪费时间，这是很致命的

whosbug 还强化了我的学习能力，不管是往宏观的快速学习，还是往微观的小细节学习，或是抓取你要用的那个小 part 来学习，这种抓 part 的能力

whosbug 也让我认识了 5 个很厉害的同学，和他们合作实话实话，不可能全是愉快的，但总体是舒服的，我也有做的不好的地方，也感谢他们的包涵

whosbug 项目只是告一段落，导师也把我们拉进了 Github 的组里，说不定还有未完待续 ~
