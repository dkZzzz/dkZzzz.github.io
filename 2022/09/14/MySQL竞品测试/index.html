MySQL竞品测试

Created2022-09-14|Updated2022-09-26
|Post View:

前言

whosbug 那边一结束，休息了两天之后，又有了一个新的项目，还是和 CSIG 有关的，做的是云服务器 MySQL 竞品测试，这次是我第一次接触测试领域，之前对测试不能说是一无所知，只能说是会简单测试，之前 whosbug 就是使用 pprof 进行测试，也是主要由我负责的，这次做的竞品测试，对比之前做的会更加正式，更加专业

具体的测试这里也不细讲，还是照旧，把项目中学到的东西，需要记录的东西给留存住，这才是我写blog 的目的

竞品测试

要做竞品测试，首先要知道什么是竞品，顾名思义，竞品就相互竞争的商品，说白了就是本质一样的东西，不同品牌，自然也对应着不同价格和不同性能等等

进行竞品测试，就拿这个云服务器 MySQL 来说，就是对比市面上在售的 MySQL，对它们进行测试， 最后得出测试报告，结论可以是哪个的性价比高，哪个的绝对性能好等等，看测试的需求

CDB vs ECS

这次做的竞品测试，是云服务器 MySQL 的，有两个大的测试对象，具体是腾讯云的 CDB 和 阿里云的 ECS，对他们使用 sysybench 进行测试，最后得出测试报告

sysbench

sysbench 是什么

sysbench 是一个开源的多线程性能测试工具，可以进行多维度的测试，具体为

  1. fileio 磁盘 io 测试
  2. cpu cpu 性能测试
  3. memory 内存分配及传输速度测试
  4. threads 线程性能测试
  5. mutex 互斥/锁性能测试
  6. oltp oltp（联机事务处理）测试

测试过程

不管是 CDB 还是 ECS，进行测试都是使用 sysbench 运行导师使用的脚本，具体细节是机器规格有两种，4core16g 和 16core64g；进行的读写类型也有两种，磁盘 io 型 和 内存型，控制这两种类型主要是由脚本里的数据量，由 tablesize 和 tablenum 来控制；机器购买的时段也有要求，这是为了考虑到云数据库高峰期超卖的情况，我们组每隔 8h 进行一次购买，然后测试，进行三组测试

Key Point

这个 part 可以是关键点，也是我学到的点（过程是遇到问题，解决问题）

内网 / 网络时延：从这个点认识到测试的严谨，因为项目需要测试的是两个品牌的 MySQL，但是还需要一个云服务器来运行 sysbench 脚本，为了保证可以 MySQL 和云服务器在同意内网下，需要购买和云数据库 MySQL 在同一可用区的云服务器，再之后远程登录云服务器 ping 通云数据库 MySQL，测试网络延迟，这个内网的延迟就会很低，但是这是一个绝对值，还需要把腾讯云的延迟同阿里云的延迟进行对比，两者差距不大的话就可以进行测试，如果相差太大，或者网络偶发性地波动很大，那么就销毁机器，重新购买

腾讯云的购买出错：从这个点也认识到可以有多种解决问题的方法，这次项目的云资源全部是以按量计费，按小时计费的方式进行购买的，腾讯云有这种小时计费实例不能同时超过10个的要求，但是由于腾讯云内部的资源回收机制，后台在销毁的小时计费实例也算在这10个里面，自己也看不见，结果导致了明面上我在使用的实例只有2个，但是之后一个也购买不了了，一开始我是想着只要时间够长，它就销毁完成了，就应该没有10个那么多，可以进行后续购买，结果从某天中午，一直等到第二天晚上，还是购买不了，已经影响到项目的进度推进了，就开始双线程找导师和咨询腾讯云客服，虽然腾讯云的客服踢皮球，但是后面还是把问题解决了，导师在之后也说，我们购买了腾讯云的东西，我们就是客户，我们遇到问题，咨询在线客服，或者是打客服电话，寻求他们的技术支持是合理的，也是解决问题的最快方法，确实是腾讯云的问题，自己研究或导师解决，很大可能是没有官方处理的快的

测试结果

最后得出的结论用一句话说就是，CDB 暂时 确实不如 ECS，但是 CDB 在多线程方面的优化，做的比 ECS 强，脚本测试了6个场景，基本上CDB 能领先的只有 1~2 个，其他都是落后，偶尔会有持平的情况出现，CDB 还是任重道远

后续

由于项目的时间比较短以及确实一开始花费在 debug 的时间比较多，（而且发现不管是 CDB 还是 ECS，都把内部的 MySQL 封装好了，不能进去进行测试，只能自己安装 MySQL 8.0，自己分析优化点）没有完成通过分析 CDB 和 ECS 测试结果的数据，使用 vtune 或 perf 进行测试，对 MySQL 的可优化点进行分析，如果要分析的话，导师给的建议的步骤是：

假设测试的结果是 CDB 相对 ECS ，只读的性能比较差，那么我们就单独测试 CDB 的只读性能，用 vtune 或者 perf 对这个进行测试，收集热点函数，观察微架构，分析火焰图，等等等等各种分析的方法，进行对应分析

导师也给了一本参考的书籍，是关于自顶向下的分析方法，比如分析 MySQL 的 cpu 性能瓶颈，需要找到瓶颈是在前端还是后端，MySQL 的前端主要是寻址功能，后端用来执行，假设瓶颈在前端，vtune 也会智能地给出建议，再对应分析热点图的前三个热点函数，对函数进行对应的优化，可以在网上搜索，也可以看看 GitHub，博客，看看别人怎么优化

如果找不出 cpu 的瓶颈的话，可以进行 MySQL 源码的阅读，分析 MySQL 的函数，对 MySQL 的函数源码进行修改，自行优化，例如循环展开，循环优化，不变量外提，循环层数降低等等，将源码变换后重新编译，然后重新启动 MySQL，再和老版本的进行性能对比
